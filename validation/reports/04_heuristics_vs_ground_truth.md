## 1. Methodology — построение ground truth

Для каждой из 200 задач бинарная метка `is_complex_gt` выведена **только из `sentence_type`** (число span'ов не используется — простое предложение может быть ошибочно разбито на span'ы, это шум разметки, а не сложность):

- `is_complex_gt = True`, если ≥2 из 3 аннотаторов поставили тип ∈ {ССП, СПП, БСП, Смешанного типа};
- `is_complex_gt = False`, если ≥2 из 3 поставили «Не сложное»;
- `ambiguous`, если большинства нет — исключается из метрик.

**Результат:** complex = 155, simple = 45, **ambiguous = 0**. Все 200 задач получили устойчивое большинство (в т.ч. 2 задачи с одним пропущенным типом — оставшиеся 2 аннотатора согласились). GT сохранён в `validation/data/ground_truth.csv`.

Эвристики (`heuristic_predictions.csv`) применены через прямой импорт `filters.py`:
- `has_conjunction(text)` — наличие союза ССП/СПП (regex по таблице `conjunctions.json`);
- `has_asyndetic_markers(text, min_tokens=8)` — длина ≥8 токенов + БСП-пунктуация делит текст на две содержательные половины;
- `count_finite_verbs(text) ≥ 2` — морфоанализ Natasha;
- `is_complex_v1 = has_conjunction OR has_asyndetic` — **используемая** версия;
- `is_complex_v2 = (count_finite_verbs ≥ 2) AND (has_conjunction OR has_asyndetic)` — отвергнутая версия (`_is_complex_v2_rejected` в `filters.py`).

Положительный класс — «сложное». Все 200 задач участвуют в метриках (ambiguous = 0).

## 4. Анализ ошибок is_complex_v1

### 4.1 False Positives — 21 (эвристика «сложное», GT «простое»)

Доминирующий паттерн — **однородные члены / однородные сказуемые**, соединённые «и» или запятыми. По правилу разметки (`notebooks/sampling/notes/edge_cases.md`) цепочка однородных членов при одном подлежащем — **одна клауза**, но `has_conjunction` срабатывает на «и», а `has_asyndetic_markers` — на запятые между однородными членами.

| # | Триггер | Текст |
|---|---|---|
| #149 | conj «и» | Я ненавижу себя **и** свою жизнь. |
| #137 | asyndetic | Я была красивая**,** жизнерадостная**,** веселая. |
| #141 | asyndetic | Я в проблемах**,** без образования**,** по уши в долгах. |
| #160 | asyndetic | Я устал видеть**,** слышать**,** чувствовать**,** думать… |
| #158 | asyndetic | Я уже так устала от этой борьбы**,** борьбы с родителями**,** борьбы с парнем… |
| #88 | conj + asyn | Но придя домой я разревелась**,** чувствуя свою ущербность. _(деепричастный оборот)_ |
| #191 | conj «почему» | **почему** в свои 14 лет я должна думать о самоубийстве! _(вопросительное слово, не союз)_ |
| #185 | conj + asyn | не любишь ты меня**,**падло:(((\*(\*:??\*::;%?\*(:\*()?97890768679 _(мусорный текст)_ |

**Паттерны FP:**
1. **Однородные члены/сказуемые** (≈14 из 21) — основной источник; «и» и запятые-перечисления. Это **известный и сознательно принятый компромисс** — `validation.md` прямо предупреждает: «A sentence … homogeneous predicates joined by «и» … will be classified as complex by V1».
2. **Причастные/деепричастные обороты** (#88, #44, #128, #21, #94) — обособляются запятыми → срабатывает `has_asyndetic`; по правилу разметки — не клауза.
3. **Вопросительно-относительное слово вне придаточного** (#191 «почему», #60) — слово из списка СПП-союзов в роли вопросительного.
4. **Мусорный/нестандартный текст** (#185) — символьный шум опознаётся пунктуационным сплиттером.

### 4.2 False Negatives — 3 (эвристика «простое», GT «сложное»)

| # | Страта | Текст |
|---|---|---|
| #14 | presuicidal_short_simple | В первом классе меня все любили всем мальчикам я нравилась. |
| #136 | presuicidal_short_simple | Я боюсь доверять людям мне недавно исполнилось 17! |
| #174 | solyanka_twitter_short_simple | везде, это полное кидалово ) мстить буду ск!) |

**Паттерн FN — единственный:** все 3 — **бессоюзные сложные предложения (БСП) без стандартной БСП-пунктуации на границе клауз**. #14 и #136 — чистые run-on'ы (граница клауз вообще без знака препинания: «…меня все любили ‖ всем мальчикам я нравилась»). #174 — граница после «)», а скобка не входит в набор БСП-маркеров `{, : — – ;}`. `has_asyndetic_markers` опирается на пунктуацию и не видит бессоюзную границу без знака. Союзов в этих текстах нет → V1 = «простое». Все 3 — из коротких страт.

## 6. Сравнение с исходной валидацией на 120 примерах Shirin

| Источник | GT | Negatives | V1 Precision | V1 Recall |
|---|---|---|---|---|
| `validation.md`, Scenario B | n_clauses>1, NaN исключены | **0** | 1.000 _(тривиально)_ | 1.000 |
| `validation.md`, Scenario A | то же + 2 plain-string | 2 | 1.000 | 1.000 |
| `validation.md`, «All» (исходный) | NaN = простое | 26 | 0.825 | 1.000 |
| **Эта валидация (200 задач)** | **majority voting по типу** | **45** | **0.879** | **0.981** |

**Где разошлось и почему.**

1. **Precision 1.0 → 0.879.** Расхождение **ожидаемо** и не является ошибкой. В Scenario B (на которую опирается заявление «P=1.0») было **ноль отрицательных примеров** — FP физически негде возникнуть, precision тривиально равна 1.0. `filters.py` это честно признаёт в docstring: «Precision could not be discriminated … Scenario B contains zero ground-truth negatives». 200-задачный GT впервые даёт 45 реальных «простых» предложений — и 21 FP становится видимым. Реальная precision (0.879) близка к нетривиальной оценке «All»-вида (0.825).
2. **Что именно стало FP.** 45 «простых» из нового GT содержат ровно те конструкции, которые `validation.md` в разделе «Known consequence» **заранее назвал**: однородные сказуемые/члены и обособленные обороты. То есть расхождение — не скрытый баг, а **уже задокументированный, сознательно принятый компромисс**, теперь впервые измеренный количественно.
3. **Recall 1.0 → 0.981 — устойчив.** Падение на 0.019 (3 FN из 155) — в пределах ожидаемого при переходе на более шумную production-выборку (короткие run-on-БСП). Заявление о near-perfect recall подтверждается.

**Вывод по разделу 6.** Заявление «V1 P=1.0, R=1.0» — **частично достоверно**. Recall ≈ 1.0 — реальное и устойчивое свойство. Precision = 1.0 — **артефакт отсутствия отрицательных примеров** в исходном тесте; на корректном GT precision = 0.879. Это не противоречит `filters.py` (его docstring точен), но табличная строка «P=1.000» в `validation.md` без оговорки о нуле негативов вводит в заблуждение.

## 7. Вывод по V1 vs V2

| Критерий | V1 (used) | V2 (rejected) | Победитель |
|---|---|---|---|
| Accuracy | 0.880 | 0.840 | V1 |
| Precision | 0.879 | 0.984 | V2 (+0.105) |
| Recall | 0.981 | 0.806 | V1 (+0.175) |
| F1 | 0.927 | 0.887 | V1 |
| FP | 21 | 2 | V2 |
| FN | 3 | 30 | V1 |

**Решение V1 не требует пересмотра.** Для задачи **отбора выборки на разметку** приоритет recall корректен: пропущенный сложный паттерн (FN) никогда не попадёт в обучающие данные и оставит слепое пятно в будущем классификаторе, тогда как FP (простое предложение в выборке) после разметки становится полезным отрицательным примером. V1 пропускает 3 сложных, V2 — 30; это решающий аргумент.

**Что изменилось по сравнению с исходным решением.** В `validation.md` преимущество V2 по precision было названо **неизмеримым** («cannot be validated … Scenario B contains zero negatives»). Теперь, на 45 реальных негативах, оно **измерено и реально**: V2 precision 0.984 против 0.879 у V1 (+0.105), gate «≥2 финитных глагола» убирает 19 из 21 FP. Но recall V2 — лишь 0.806 (30 FN), и провал по recall (−0.175) втрое перевешивает выигрыш по precision. Исходный вердикт «recall priority → V1» подтверждён и теперь подкреплён числами, а не предположением.

**Итог:** V1 остаётся доминирующим. Пересмотр не нужен. Точечная доработка — не замена V1 на V2, а устранение конкретных слепых зон V1 (см. ниже).

## Рекомендации для следующего цикла сэмплирования

1. **Фильтр однородных членов/сказуемых** — снимет основной класс FP (≈14 из 21). Кандидат: не считать запятые-перечисления при общем подлежащем БСП-маркером; не считать одиночное «и» между однородными членами союзом ССП. Особенно важно для страты `presuicidal_multilabel_bsp` (precision 0.714).
2. **Детектор бессоюзных run-on'ов без пунктуации** — снимет все 3 FN. `has_asyndetic_markers` сейчас завязан на пунктуацию; нужен сигнал по числу финитных предикатов / подлежащих без знаков препинания, особенно для коротких разговорных страт.
3. **Не доверять precision = 1.0 без проверки числа негативов** — любая будущая валидация эвристики должна явно указывать `n_negatives`; при нуле негативов precision не информативна.
